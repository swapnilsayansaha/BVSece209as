Real-time Implementation

For simulation, we attempted to implement virtual networks in Mininet (shown in Figure below).

  • Inodes are basically the input nodes - these are the sources of sensor data.
  • ​Cnodes are the computation nodes - they perform computations over the received sensor data and also communicate with each other in the case of SMPC.
  • All the other squares (i.e. i1s, c1s, s123…) are network switches that just route traffic.
  • The naming convention is ‘i’ for inode, ‘c’ for cnode, and ‘s’ for switch, and the next number indicates which node/switch it is connected to, e.g. i1s is a switch connected to inode 1. ​s123 is a switch connected to switches i1s,i2s,i3s.

​The topology was made from a tool called Spear (http://demo.spear.narmox.com/app/?apiurl=demo#!/mininet).
Figure (Left): Test network implemented in Mininet. (Right): Snippets of packet logs being exchanged among nodes.
For hardware implementation, we chose Raspberry Pi 4 (CPU – Broadcom BCM2711, Quad core Cortex-A72 (ARM v8) 64-bit SoC @ 1.5GHz, 4 GB RAM, 32 GB flash storage and Gigabit Ethernet) running on 64-bit Ubuntu 18.04.4 LTS and compiled the Sharemind SDK locally. However, we could only benchmark secure aggregation protocols in networked implementation on Mininet as Sharemind requires a license to operate. However, the protocols are not closed-source, hence, we managed to benchmark the ABY protocol for SMPC.

Benchmarking on Mininet / Raspberry Pi

For stand-alone secure aggregation demo, we benchmarked key generation, oblivious add/multiply, encryption and decryption operations on Raspberry Pi 4.
  • CPU usage of key generation: 99.7% on Broadcom BCM2711, Quad core Cortex-A72 (ARM v8) 64-bit SoC @ 1.5GHz
  • RAM consumption of key generation: 0.8-0.9% of 4 GB (33 MB - 36 MB)
  • Time taken for key generation: 105.661155065 mS (average from 1000 key generations)
  • Time taken for encryption: 40.015435095 mS
  • Time taken for decryption: 12.323417938 mS
  • Time Elapsed for scalar addition: 336.331981 nS​
  • Time Elapsed for scalar multiplication: 374.014662 nS​

We observe the following from our benchmark:
  • Secure aggregation is more suited to time-sensitive/critical computations in real-time such as queries, that do not require large amount of resources.
  • Secure MPC is more efficient in dealing with offline computations, such as classification/regression analysis on joint dataset, that does require some amount of resources.
  • Network latency is negligible for both architectures. We assume the nodes are connected with GBPS ethernet. Majority of the latency is contributed to by the computation time only.
  • While Shared3p/ABY offers better performance than SPDZ, they are not secure against malicious adversaries.
