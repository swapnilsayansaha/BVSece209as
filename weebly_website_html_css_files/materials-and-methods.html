Materials and Methods

Personal data, particularly medical data, has incredible utility yet entails many privacy concerns. When external entities wish to analyze patient data as a group, we need methods of securely aggregating patient data to draw important inferences such as counting the number of patients infected with a particular disease, creating collaborative inferencing systems between different hospital databases without revealing patient data and identities. For example, the World Health Organization (WHO) may want to know the global infection rate of a particular disease, yet each country involved may want to keep their own nationwide infection numbers private. The notion here is that a healthcare provider/party wishes to contribute some of their data for a joint inference/analysis yet covets to keep their patients’ data private. Secure multiparty computation (SMPC) and secure aggregation come into play in such scenarios, making it possible to perform calculations on encrypted data without revealing the inputs. We seek to evaluate the two different methods across performance, privacy and security metrics to illustrate the conditions where we may choose one over the other in these collaborative systems, with the ultimate goal of creating a real-time, robust and scalable privacy preserving secure inferencing system at the edge, void of the notion of any external trusted-third parties (TTP).

Secure Aggregation

In secure aggregation, input data for computation is sent from data nodes (S) to a central node called the aggregator in homomorphic form. The aggregator is also responsible for receiving and processing homomorphic queries from query nodes (Q). The aggregator computes requested statistic on the homomorphic data and sends the results back to the query nodes. Neither the aggregator nor the query/data nodes are able to see each other's datastores or actual queries and the aggregator also acts as a middleman, masking the identities of the participating parties from each other. 
Figure: Conceptualized secure aggregation architecture for star-exchange MCPS using Paillier cryptosystem.
For the attack model, we assume that all the participating nodes in the network are honest but curious (semi-honest). The aggregator is a semi-honest device that cannot delete or ignore data from certain nodes and is honest in dealing with the query and performs exactly what the query node desires. We also assume that the adversary wants to eavesdrop on any available data in the network with highest priority on the raw sensor node data. The adversary can be present anywhere in the network and may also can try to sense the statistic and functions from the aggregator. The work can be further extended by having an adversarial aggregator. In this case, we can have two separate messages from each data-query node pair and for one of those messages, the keys are provided by the aggregator. Another strategy might be to establish anonymized direct communication among the nodes and exchange some messages to authenticate the aggregator using zero-knowledge proofs.

The proposed secure aggregation architecture has a two-layer security protocol over which it operates; the nodes and the aggregators have a layer of Advanced Encryption Standard (AES) encryption to ensure nodes or adversaries cannot overhear each other’s data. Each node has an AES key pair that is provided by the aggregator. The actuator node could, in theory operate on a polynomial-type function (e.g. Shamir's Secret Sharing) of the sensor data from various data nodes, but cannot use this data directly due to following reasons:

  1. The connection between query nodes and data nodes costs more in terms of communication parameters like power and bandwidth.
  2. ​It is not feasible for the query node to combine the huge data if the query node is also a low end device. 
​
We use a homomorphic encryption scheme called Paillier homomorphic encryption which allows homomorphic additions and multiplications. Hence, the aggregator, despite being able to obliviously compute over the homomorphic data, never sees the raw sensor data and the query node never receives the raw sensor data in any case. This also ensures that sensitive information about one particular node is not revealed to the aggregator or query or other data nodes. We assume all nodes have the same homomorphic key-pairs but different obfuscation random value. The aggregator here does not refer to any TTP (e.g. cloud services), but rather refers to a trusted server entity within the collaborating organizations (e.g. hospitals).

We have used Python for implementation of our secure aggregation architecture. For Paillier encryption. we used the PHE library which does the standard Pailier Homomorphic Encryption. First, a pair of keys is generated. Both of the keys are available to all the clients  but no key is available to the server, which can only evaluate on the encrypted/homomorphic data and pass the statstic to the query nodes. 

For preliminary benchmarking, addition and multiplication are performed 10000 times for timing as the operations are extremely fast over the same numbers (on the order of nanoseconds). In order to account for randomness, we run the addition and multiplication for 100 iterations and each time a random number is chosen in range from 0 to 256 (integer datatype). In order to inspect the memory the task was run multiple times and memory was inspected using task/activity manager. Four oblivious functions: mean, vector sum, linear regression and convolution were implemented using the homomorphic addition and multiplication operations.

The secure aggregation scheme was also implemented in Mininet for benchmarking performance metrics relevant to networked environments.

Secure Multiparty Computation

In SMPC, there is no notion of centralized entity. Rather, every node (or k nodes) participate distributively in the oblivious computation on homomorphic data and floods the network with fractional results, which are then combined by the appropriate query node using appropriate SMPC scheme to obtain the statistic. Every node may virtually act as "data", "query" and "aggregator" nodes from secure aggregation.
Figure: Conceptualized SMPC architecture for MCPS.
The attack model is similar to the secure aggregation scheme, assuming all the participating datastores are semi-honest (and can be extended to be adverserial). Zero-knowledge proofs are used for authentication of node identity and computed statistic.

When a particular node requests some particular statistic/query, the query of the node is flooded homomorphically throughout the entire network. The SMPC protocol SDK running on the machines then randomly chooses k nodes to perform the aggregation and computation. If a particular node receives computation requests, then it requests appropriate homomorphic data by flooding the network with data request query. The appropriate datastores supply data in homomorphic form (identities hidden) to the computation nodes distributively, such that one node does not receive all of the data. The query/statistic is evaluated distributively and the fractional results are flooded back throughout the network, captured by the query node. By sharing the computational power distributively, the following are achieved:
​
  1. The computing requirement is drastically reduced.
  2. One node does not learn too much about the data as the data is chunked (along with the computation) 

Similar to secure aggregation architecture, the SMPC architecture also features a two-layer security: AES over homomorphic encryption. For proposed architecture, we have chosen to evaluate the following SMPC protocols with homomorphic addition and multiplication:
​
  1. ABY: Arithmetic, binary and Yao 3PC - 3PC with secret sharing for privacy preserving machine learning and database joins (PSI, Union, etc.); secure against semi-honest adversaries; randomly goes back and forith between A, B and Y.
  2. SPDZ (Fresco): MPC with garbled circuits or secret sharing; secure against malicious or semi-honest adversaries with dishonest or honest majority. SPDZ denotes a MPC scheme in the pre-processing model based on somewhat homomorphic encryption (SHE) in the form of BGV.
  3. Shared 3P: 3PC with secret sharing; secure against semi-honest adversaries and uses the additive secret sharing scheme in the ring Z2(32).
 
For preliminary implementation of SMPC protocols, we chose to use ShareMind SDK, a prototyping emulator platform for benchmarking and evaluating SMPC algorithms, coded using SecreC 2 programming language. Preliminary benchmarking was done in a Raspberry Pi 4 style resource-constrained virtual environment for scalar addition and multiplication on homomorphic data. Real-time benchmarking in a networked environment was done on two Raspberry Pi 4's connected via ethernet running the ShareMind framework, as well as simulation in Mininet.

The success metrics of the project are:
  • Low latency and execution time of PPI system in resource-constrained setting.
  • Integrability of code with existing system.
  • No compromise in security for performance.
  • 0 need for trusted third party.
